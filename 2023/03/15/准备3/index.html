什么是spring AOP

aop 是面向切面编程，他是对面向对象编程的补充，为什么有他呢，因为在编程过程中，会有一些不是核心业务逻辑所关心的，但是有必须有的一段代码，而且是很多业务都需要的，所以为了使程序员能够着重关注核心业务逻辑以及增加了代码复用，所以就有了aop,应用场景就是日志、鉴权等，那么他是怎么实现的呢，他将这部分共用的代码封装为一个切面，然后织入到对应需要的切入点，他是基于代理实现的，spring 支持的是jdk动态代理和cglib动态代理，jdk需要实现接口，cglib’不需要实现接口，他是基于ASM原理，他里面的组成部分，首先定义一个切点aspect,然后通过pointCut通过切入表达式去定义哪些需要切入，这里就有两种，一种是基于路径，一种是基于注解，然后有5种通知，前置、返回后，异常后、后置、坏绕，环绕可以通过jonitpoint去实现这这几种，还可以通过jonitpoint.getsignature去获取该方法的方法名等

自定义注解：target 可以注解到哪些上面，retention在什么时候织入

@Slf4j
@Aspect
@Component
public class ControllerAop {

    // 切入点表达式：用来指定哪些类需要代理
    @Pointcut("execution(* com.balloon..controller.*(..))")
    public void controllerPointcut() {
    }

    @Around("controllerPointcut()")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        boolean hasException = false;
        String method = joinPoint.getSignature().toShortString();
        Result<?> result = null;
        try {
            Object obj = joinPoint.proceed();
            if (obj instanceof Result) {
                result = (Result<?>) obj;
            }
            return obj;
        } catch (Throwable e) {
            log.error("[Controller] {}执行异常, param={}", method, JSONUtil.toJSONString(joinPoint.getArgs()), e);
            hasException = true;
            throw e;
        } finally {
            String cost = String.valueOf(System.currentTimeMillis() - start);
            String suc = "-", code = "-", msg = "-";
            if (result != null) {
                suc = String.valueOf(Result.isSuccess(result));
                code = String.valueOf(result.getCode());
                msg = result.getMsg();
            }
            log.info("[Controller] {}执行结束, 耗时{}ms, exception={}, suc={}, code={}, msg={}", method, cost, hasException, suc,
                    code, msg);
        }
    }

}

负载均衡的策略

负载均衡是什么：负载均衡就是请求分发到同一个服务的不同服务器节点上，防止因为大量请求同时访问一个服务器使服务器宕机。

负载均衡的策略有：

  • 轮询：依次将请求分配给后端服务器：缺点：不适合长连接以及性能不同的服务器应该分配的请求不用
  • 加权：每个节点轮询的个数不同
  • 最小响应时间：把请求分给平均响应时间最短的后端服务器，平均响应时间可以通过ping去探测请求
  • 最小连接数：把请求分给活动连接数最小的后端服务器，但是需要维护后端服务器的连接列表
  • hash:包括对Source IP、url等的hash,

负载均衡分为服务端负载均衡和客户端负载均衡，先通过zookeeper根据serviceName去获得节点的所有urllist,然后通过一致性hash去获得某个具体的url

一致性hash

一致性hash解决了普通hash的问题，第一个就是如果一个节点宕机了，那么所有请求就会重新hash，然后又有可能造成某一个节点流量过大，造成雪崩效应，第二个问题就是扩容也会导致所有的请求都重新hash。

所以一致性hash就解决了这两个问题，他是通过hash环来解决的，那么hash环也有一个问题就是有可能使分布不均（数据倾斜），所以就有了虚拟节点， 每个物理节点都虚拟出多个虚拟节点，这样的话就避免了hash数据倾斜的问题

接口和抽象类以及区别

接口和抽象类都是为了面向对象OOP编程思想的抽象，他们都是对一个实体类的抽象

区别：

1：从设计的本身出发：接口更多的是一种行为规范，是一种功能上的共性，而抽象类更多的是一种模板式设计，是属性上的共性，就比如门有开和关两个方法，但是有的门有报警器，但有的门没有，所以这里不能直接把开和关和报警放在同一个接口或者抽象类里面，而是开和关是所有门的属性，所以需要设计一个抽象类，然后报警是个功能，所以设计为接口

2.第二就是语法和使用上的区别：接口的话因为是行为规范，所以一个类可以有多个行为，那么他就可以实现多个接口，然后抽象类因为是属性上的，所以只能单继承；在具体的语法上：接口不能实现方法，除了default方法，所以一旦实现了某个接口就需要重写所有的方法，而抽象类因为是模板式设计，所以他可以有自己的普通方法，子类可以直接调用

GC什么时候发生

GC分为minnr GC和full GC,minor GC是对新生代中的垃圾回收，当新生代中的eden区不够分配对象的时候，就会发生minor GC，minor GC采用而是复制算法，

full GC发生时期，当老年代的内存不够容不下对象的时候，就发生full GC,一般有两种情况，就是在minnor GC的时候，会首先有一个老年代担保原则，老年代剩余的可用的连续空间如果小于新生代历代minor GC升入老年代对象的总和的平均大小，就会发生一次full GC。还有就是当minnor GC之后，如果晋升的对象老年代放不下，也会full GC,

full GC可以回收方法区中的无用的常量和无用的类，无用的类的判定：没有实例对象、类加载器被回收、class对象不再任何地方被访问，以及无法通过反射访问该类，就允许被回收，但是一般方法区中的回收性价比是很低的，所以的话，可以设置虚拟机参数来确定是否回收

session、cookie、token和jwt

因为http是无状态的协议，所以每次请求都无法知道客户端的身份，那么如果每次都登陆校验，肯定不可能，所以就有了session，第一次用用户名和密码登陆的时候，服务器端就生成一个session,存储在服务器端，然后将sessionID返回给客户端，客户端将sessionID以cookie的方式存储在客户端本地，然后以后每次访问的时候，都带着该cookie,服务器就可以快速知道身份。

讲讲悲观锁和乐观锁的区别

悲观锁比较悲观，就是指多线程同时修改共享资源的概率比较高，很容易出现冲突，所以在访问共享资源前，就加上锁

而乐观锁的工作方式就是:先修改共享资源，然后再写回内存的时候，去验证在修改这段时间内，该资源是否被其他线程修改过，没有的话，就完成操作，要是被其他线程修改过的话，就一般配套自旋锁去自旋获取锁，所以一般乐观锁用于冲突概率较低的。

CAS是乐观锁，基于CAS实现的自旋锁就是悲观锁了，因为自旋锁是必须拿到锁才能去修改资源，所以乐观锁和悲观锁最本质的区别就在于先拿锁在修改资源还是不用拿锁就可以修改资源

谈谈互斥锁和自旋锁

互斥锁就是没拿到锁就释放cpu,将线程状态变为阻塞，然后这期间就有两次的用户态和内核态的上下文切换，

而自旋锁就是没有拿到锁之后，将线程还是运行，在短暂时间内再去获取，这样就避免了线程上下文切换。

自旋锁一般用于我们知道锁住的资源很快就会被释放，不然的话就会浪费CPU

唯一索引和普通索引在读和写的性能上有没有什么差异

结论：在读方面的话，性能差别不大，在写的性能上就有一些差异

​ 在读的时候，扫描索引的时候，扫描到叶子节点之后，唯一索引就不需要继续扫描了，因为唯一性的原因，而普通索引还需要继续往下扫描数据，直到条件不符合的数据，但是因为mysql在读取数据时是读取页，一页里面也有很多行，所以一般来说，不需要再次磁盘IO,除非扫描到的数据正好是数据页的最后一行，那么就需要多一次IO去读取下一页，所以在读的时候性能差别不大。

​ 在了解写的性能差异之前，首先先了解一下change buffer这个东西。在我们读的时候，如果数据页在内存当中的话，就可以直接返回不用每次都去读取磁盘IO,那在写的时候我们也有一套类似的机制，mysql在来了更改的请求时，更改内存，然后将更改记录在change buffer中，然后以redo log的方式持久化（刷盘的几种机制）。

​ 在普通索引的时候，写请求来了之后，如果数据页不在内存中，就将写的操作记录在change buffer中，当需要读取该页时，即需要去读取磁盘的时候，读取完之后就将change buffer中操作更新在数据页上（merge），那么就减少了之前更新需要读取的磁盘IO，那么在唯一性索引的时候，因为需要判断是否是唯一的，所以每次写的时候都需要去读取磁盘IO，所以就多了一次IO。

对于选择唯一索引还是普通索引还是得根据业务场景来，如果是写多读少，例如财务或者日志类就可以选择普通索引来提高性能，对于一般的写少读多的，写了之后就要读的话，那么有了change buffer反而增加了一层change buffer 的维护，所以这种场景就选择唯一索引。

LRU和LFU算法（算法实现在准备1）

LRU淘汰最近没有使用的，LFU淘汰访问频率最低的，如果访问频率一致，就淘汰最久的。

这里需要注意的是，LFU的访问频率怎么计算，这里主要讲讲redis中的LFU淘汰算法中的访问频率：

在redis 中，键值对的值是一个redisObject,在redisObject中，有一个24bits的lru变量，如果实在LRU算法中，所以就直接以该变量来存储该键最近一次访问时间，然后通过这个来排序淘汰键；如果在lfu中，低8bit表示访问次数，高16bit表示最近一次访问时间。那么键值对的初始化以及更新是怎么更新的，以及淘汰怎么淘汰的

键值对的初始化：时间戳是以分钟为精度的，访问次数是指数的大小，而又因为避免新数据直接被淘汰，所以初始化为5

键值对被访问时

第一步：根据距离上次访问的时长，衰减访问次数。

主函数：updateLFU会首先调用LFUDecrAndReturn函数去衰减访问次数，LFUDecrAndReturn函数的逻辑是，首先获取lru变量的高16bit去获取该键上一次访问的时间，然后会去获取全局变量server里面的lru_decay_time ，去计算需要衰减的访问次数，计算过程就是：就会去调用LFUTimeElapsed，去计算上距离上一次访问的时长，然后除以lru_decay_time，就是需要衰减的次数。

（Redis 在初始化时，会通过 initServerConfig 函数来设置 lfu_decay_time 变量的值，默认值为 1。所以，在默认情况下，访问次数的衰减大小就是等于上一次访问距离当前的分钟数。比如，假设上一次访问是 10 分钟前，那么在默认情况下，访问次数的衰减大小就等于 10

第二步：根据当前访问更新访问次数：

如果当前访问次数大于255（8位），LFULogIncr函数不再增加访问次数。访问次数小于255，LFULogIncr就会计算出一个阈值p，以及一个0-1的随机值r，如果r<p, LFULogIncr 函数才会将访问次数加 1。否则的话，LFULogIncr 函数会返回当前的访问次数，不做更新。

从这里你可以看到，因为概率值 r 是随机定的，所以，阈值 p 的大小就决定了访问次数增加的难度。阈值 p 越小，概率值 r 小于 p 的可能性也越小，此时，访问次数也越难增加；相反，如果阈值 p 越大，概率值 r 小于 p 的可能性就越大，访问次数就越容易增加。

而阈值 p 的值大小，其实是由两个因素决定的。一个是当前访问次数和宏定义 LFU_INIT_VAL 的差值 baseval，另一个是 redis.conf 文件中定义的配置项 lfu-log-factor。

当计算阈值 p 时，我们是把 baseval 和 lfu-log-factor 乘积后，加上 1，然后再取其倒数。所以，baseval 或者 lfu-log-factor 越大，那么其倒数就越小，也就是阈值 p 就越小；反之，阈值 p 就越大。也就是说，这里其实就对应了两种影响因素。

baseval 的大小：这反映了当前访问次数的多少。比如，访问次数越多的键值对，它的访问次数再增加的难度就会越大；
lfu-log-factor 的大小：这是可以被设置的。也就是说，Redis 源码提供了让我们人为调节访问次数增加难度的方法

为什么需要阈值：为了低8位的表示的数有限。

最后，到这一步，updateLFU 函数已经完成了键值对访问次数的更新。接着，它就会调用 LFUGetTimeInMinutes 函数，来获取当前的时间戳，并和更新后的访问次数组合，形成最新的访问频率信息，赋值给键值对的 lru 变量。

LFU算法淘汰的逻辑：和LRU差不多

因为常规的LRU会让LFU需要有一个缓存容量的链表来保存数据，然后每次访问都需要去操作该链表，所以在redis中的数据缓存的比较多的时候，就额外需要一个内存空间来保存链表（如果需要降低时间复杂度的话，就还需要一个hashmap来保存数据），如果不需要降低时间复杂度的话，时间消耗就较大，为了保持redis的高性能，redis源码没有严格实现LRU和LFU的淘汰算法，而是近似算法，

第一步：从待采样的哈希表中随机获取一定数量的 key。如果 maxmemory_policy 配置的是 allkeys_lru，那么待采样哈希表就是 Redis server 的全局哈希表，也就是在所有键值对中进行采样；否则，待采样哈希表就是保存着设置了过期时间的 key 的哈希表。函数采样的 key 的数量，是由 redis.conf 中的配置项 maxmemory-samples 决定的，该配置项的默认值是 5。
第二步：尝试把采样的每一个键值对插入 EvictionPoolLRU 数组。这主要取决于以下两个条件之一：一是，它能在数组中找到一个尚未插入键值对的空位；二是，它能在数组中找到一个空闲时间小于采样键值对空闲时间的键值对。这两个条件有一个成立的话，就会把采样的键值对插入 EvictionPoolLRU 数组。
第三步：选择被淘汰的键值对并删除。遍历 EvictionPoolLRU 数组（这个数组里面的 key，是按照空闲时间从小到大排好序了），从数组的最后一个 key 开始选择，如果选到的 key 不是空值，那么就把它作为最终淘汰的 key。
第四步：一旦选到了被淘汰的 key，就会根据 Redis server 的惰性删除配置，来执行同步删除或异步删除
上面四步是一个循环遍历的过程，也就是说一旦触发了LRU淘汰，如果没有达到我前面介绍的待释放空间，会重复执行上面的四步，直到满足待释放空间的大小要求。

其实，我们可以看到，近似 LRU 算法并没有使用耗时耗空间的链表，而是使用了固定大小的待淘汰数据集合，每次随机选择一些 key 加入待淘汰数据集合中。最后，再按照待淘汰集合中 key 的空闲时间长度，删除空闲时间最长的 key。这样一来，Redis 就近似实现了 LRU 算法的效果了。

第四步淘汰数据总结
实现一个严格的 LRU 算法，需要额外的内存构建 LRU 链表，同时维护链表也存在性能开销，Redis 对于内存资源和性能要求极高，所以没有采用严格 LRU 算法，而是采用「近似」LRU 算法实现数据淘汰策略
触发数据淘汰的时机，是每次处理「请求」时判断的。也就是说，执行一个命令之前，首先要判断实例内存是否达到 maxmemory，是的话则先执行数据淘汰，再执行具体的命令
淘汰数据时，会「持续」判断 Redis 内存是否下降到了 maxmemory 以下，不满足的话会继续淘汰数据，直到内存下降到 maxmemory 之下才会停止
可见，如果发生大量淘汰的情况，那么处理客户端请求就会发生「延迟」，影响性能
Redis 计算实例内存时，不会把「主从复制」的缓冲区计算在内，也就是说不管一个实例后面挂了多少个从库，主库不会把主从复制所需的「缓冲区」内存，计算到实例内存中，即这部分内存增加，不会对数据淘汰产生影响

总结

到这里，你就了解了，Redis 源码在更新键值对访问频率时，对于访问次数，它是先按照上次访问距离当前的时长，来对访问次数进行衰减。然后，再按照一定概率增加访问次数。这样的设计方法，就既包含了访问的时间段对访问频率的影响，也避免了 8 bits 计数器对访问次数的影响。而对于访问时间来说，Redis 还会获取最新访问时间戳并更新到 lru 变量中。

为什么数据库不建议使用外键

外键提供的几种在更新和删除时的不同行为都可以帮助我们保证数据库中数据的一致性和引用合法性，但是外键的使用也需要数据库承担额外的开销，在大多数服务都可以水平扩容的今天，高并发场景中使用外键确实会影响服务的吞吐量上限。在数据库之外手动实现外键的功能是可能的，但是却会带来很多维护上的成本或者需要我们在数据一致性上做出一些妥协。我们可以从可用性、一致性几个方面分析使用外键、模拟外键以及不使用外键的差异：

  • 不使用外键牺牲了数据库中数据的一致性，但是却能够减少数据库的负载；
  • 模拟外键将一部分工作移到了数据库之外，我们可能需要放弃一部分一致性以获得更高的可用性，但是为了这部分可用性，我们会付出更多的研发与维护成本，也增加了与数据库之间的网络通信次数；
  • 使用外键保证了数据库中数据的一致性，也将全部的计算任务全部交给了数据库；

在大多数不需要高并发或者对一致性有较强要求的系统中，我们可以直接使用数据库提供的外键帮助我们对数据进行校验，但是在对一致性要求不高的、复杂的场景或者大规模的团队中，不使用外键也确实可以为数据库减负，而大团队也有更多的时间和精力去设计其他的方案，例如：分布式的关系型数据库。

当计算阈值 p 时，我们是把 baseval 和 lfu-log-factor 乘积后，加上 1，然后再取其倒数。所以，baseval 或者 lfu-log-factor 越大，那么其倒数就越小，也就是阈值 p 就越小；反之，阈值 p 就越大。也就是说，这里其实就对应了两种影响因素。

baseval 的大小：这反映了当前访问次数的多少。比如，访问次数越多的键值对，它的访问次数再增加的难度就会越大；
lfu-log-factor 的大小：这是可以被设置的。也就是说，Redis 源码提供了让我们人为调节访问次数增加难度的方法。

LFU算法淘汰的逻辑：和LRU差不多

因为常规的LRU会让LFU需要有一个缓存容量的链表来保存数据，然后每次访问都需要去操作该链表，所以在redis中的数据缓存的比较多的时候，就额外需要一个内存空间来保存链表（如果需要降低时间复杂度的话，就还需要一个hashmap来保存数据），如果不需要降低时间复杂度的话，时间消耗就较大，为了保持redis的高性能，redis源码没有严格实现LRU和LFU的淘汰算法，而是近似算法，

第一步：从待采样的哈希表中随机获取一定数量的 key。如果 maxmemory_policy 配置的是 allkeys_lru，那么待采样哈希表就是 Redis server 的全局哈希表，也就是在所有键值对中进行采样；否则，待采样哈希表就是保存着设置了过期时间的 key 的哈希表。函数采样的 key 的数量，是由 redis.conf 中的配置项 maxmemory-samples 决定的，该配置项的默认值是 5。
第二步：尝试把采样的每一个键值对插入 EvictionPoolLRU 数组。这主要取决于以下两个条件之一：一是，它能在数组中找到一个尚未插入键值对的空位；二是，它能在数组中找到一个空闲时间小于采样键值对空闲时间的键值对。这两个条件有一个成立的话，就会把采样的键值对插入 EvictionPoolLRU 数组。
第三步：选择被淘汰的键值对并删除。遍历 EvictionPoolLRU 数组（这个数组里面的 key，是按照空闲时间从小到大排好序了），从数组的最后一个 key 开始选择，如果选到的 key 不是空值，那么就把它作为最终淘汰的 key。
第四步：一旦选到了被淘汰的 key，就会根据 Redis server 的惰性删除配置，来执行同步删除或异步删除
上面四步是一个循环遍历的过程，也就是说一旦触发了LRU淘汰，如果没有达到我前面介绍的待释放空间，会重复执行上面的四步，直到满足待释放空间的大小要求。

其实，我们可以看到，近似 LRU 算法并没有使用耗时耗空间的链表，而是使用了固定大小的待淘汰数据集合，每次随机选择一些 key 加入待淘汰数据集合中。最后，再按照待淘汰集合中 key 的空闲时间长度，删除空闲时间最长的 key。这样一来，Redis 就近似实现了 LRU 算法的效果了。

第四步淘汰数据总结
实现一个严格的 LRU 算法，需要额外的内存构建 LRU 链表，同时维护链表也存在性能开销，Redis 对于内存资源和性能要求极高，所以没有采用严格 LRU 算法，而是采用「近似」LRU 算法实现数据淘汰策略
触发数据淘汰的时机，是每次处理「请求」时判断的。也就是说，执行一个命令之前，首先要判断实例内存是否达到 maxmemory，是的话则先执行数据淘汰，再执行具体的命令
淘汰数据时，会「持续」判断 Redis 内存是否下降到了 maxmemory 以下，不满足的话会继续淘汰数据，直到内存下降到 maxmemory 之下才会停止
可见，如果发生大量淘汰的情况，那么处理客户端请求就会发生「延迟」，影响性能
Redis 计算实例内存时，不会把「主从复制」的缓冲区计算在内，也就是说不管一个实例后面挂了多少个从库，主库不会把主从复制所需的「缓冲区」内存，计算到实例内存中，即这部分内存增加，不会对数据淘汰产生影响

AQS核心思想：

这样性能很低下，因为一个数据页是16k,而如果每次修改几个字节都将整个数据页刷新的话，性能肯定不好，所以这里采用了WAL技术，写入日志，有几个好处：顺序读写，而且一行日志只有几十个字节，在redo log 写满之后还会有合并刷盘，效率更高

AOF

AOF是append采用追加的方式，没操作一次写命令，都将执行的命令写入到aof中。

过程：先执行操作，然后将其命令记录到操作日志中，这样做的好处就是减少了命令的额外检查以及不会阻塞当前操作

（1）刷盘策略

写入aof时，是先将命令写入到aof缓冲区中，然后通过系统调用，将aof_buf缓冲区的数据写到内核缓冲区中，等待内核将其刷盘，这里可以通过刷盘机制控制（调用fsync函数），always、EverySec、no，根据系统的考虑选择

（2）AOF后台重写

因为AOF是记录操作命令的，所以文件很快就会很大，所以就有了aof重写，当文件大于64mb的时候，就会出发aof重写，重写就是为了将作为历史命令的记录去除，但是是重新写一个aof文件，然后写完之后再覆盖新的文件，

aof写入命令的操作是主进程操作的，但是aof重写还是比较耗时的，所以不能用主进程，而是用了后台子进程，这样的话，就不会阻塞主进程，那么子进程是怎么共享父进程的数据的呢，这里就涉及到操作系统的知识，子进程fork父进程的页表，然后权限是只读，但是如果发生了写操作（写时复制技术），就会拷贝物理地址，

那在子进程重写aof的时候，主进程写的增量数据怎么办，这里就用了一个aof缓冲区，主进程将写的命令写在aof缓冲区，然后等待子进程重写完之后，再将缓冲区写入新的aof中

在面向对象的编程中，程序的核心概念是对象。对象是具有状态（属性或数据）和行为（方法或函数）的实体。通过定义类，我们可以创建多个对象，每个对象都有自己的状态和行为，并且可以与其他对象进行交互。

面向对象的编程具有以下主要特点：

  1. 封装（Encapsulation）：封装是将数据和相关操作封装在一个单元（类）中的概念。对象的内部状态对外部是不可见的，只有通过公共接口（方法）才能访问和操作对象的状态。这提供了信息隐藏和隔离的机制，使得对象的内部实现对外部是透明的。
  2. 继承（Inheritance）：继承是面向对象编程中的一种机制，它允许创建新类时基于现有类的属性和行为进行扩展。通过继承，子类可以继承父类的属性和方法，并可以在此基础上进行修改或扩展。继承促进了代码的重用和层次化设计。
  3. 多态（Polymorphism）：多态是指同一类型的对象在不同的上下文中可以表现出不同的行为。通过多态，可以使用父类类型引用来引用子类对象，并根据实际对象类型的不同来调用相应的方法。多态提高了代码的灵活性和可扩展性。

面向对象的编程方法有助于组织复杂的系统，并提供了更高的可维护性、可扩展性和可重用性。它通过将程序分解为对象，将问题分解为类和对象之间的交互，使得代码更易于理解、修改和扩展。同时，面向对象的编程还能够更好地模拟现实世界的问题和概念，提供了一种直观和自然的编程模型。
