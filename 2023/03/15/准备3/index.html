什么是spring AOP

aop 是面向切面编程，他是对面向对象编程的补充，为什么有他呢，因为在编程过程中，会有一些不是核心业务逻辑所关心的，但是有必须有的一段代码，而且是很多业务都需要的，所以为了使程序员能够着重关注核心业务逻辑以及增加了代码复用，所以就有了aop,应用场景就是日志、鉴权等，那么他是怎么实现的呢，他将这部分共用的代码封装为一个切面，然后织入到对应需要的切入点，他是基于代理实现的，spring 支持的是jdk动态代理和cglib动态代理，jdk需要实现接口，cglib’不需要实现接口，他是基于ASM原理，他里面的组成部分，首先定义一个切点aspect,然后通过pointCut通过切入表达式去定义哪些需要切入，这里就有两种，一种是基于路径，一种是基于注解，然后有5种通知，前置、返回后，异常后、后置、坏绕，环绕可以通过jonitpoint去实现这这几种，还可以通过jonitpoint.getsignature去获取该方法的方法名等

自定义注解：target 可以注解到哪些上面，retention在什么时候织入

@Slf4j
@Aspect
@Component
public class ControllerAop {

    // 切入点表达式：用来指定哪些类需要代理
    @Pointcut("execution(* com.balloon..controller.*(..))")
    public void controllerPointcut() {
    }

    @Around("controllerPointcut()")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        boolean hasException = false;
        String method = joinPoint.getSignature().toShortString();
        Result<?> result = null;
        try {
            Object obj = joinPoint.proceed();
            if (obj instanceof Result) {
                result = (Result<?>) obj;
            }
            return obj;
        } catch (Throwable e) {
            log.error("[Controller] {}执行异常, param={}", method, JSONUtil.toJSONString(joinPoint.getArgs()), e);
            hasException = true;
            throw e;
        } finally {
            String cost = String.valueOf(System.currentTimeMillis() - start);
            String suc = "-", code = "-", msg = "-";
            if (result != null) {
                suc = String.valueOf(Result.isSuccess(result));
                code = String.valueOf(result.getCode());
                msg = result.getMsg();
            }
            log.info("[Controller] {}执行结束, 耗时{}ms, exception={}, suc={}, code={}, msg={}", method, cost, hasException, suc,
                    code, msg);
        }
    }

}

day 32 3.19

负载均衡的策略

负载均衡是什么：负载均衡就是请求分发到同一个服务的不同服务器节点上，防止因为大量请求同时访问一个服务器使服务器宕机。

负载均衡的策略有：

  • 轮询：依次将请求分配给后端服务器：缺点：不适合长连接以及性能不同的服务器应该分配的请求不用
  • 加权：每个节点轮询的个数不同
  • 最小响应时间：把请求分给平均响应时间最短的后端服务器，平均响应时间可以通过ping去探测请求
  • 最小连接数：把请求分给活动连接数最小的后端服务器，但是需要维护后端服务器的连接列表
  • hash:包括对Source IP、url等的hash,

负载均衡分为服务端负载均衡和客户端负载均衡，先通过zookeeper根据serviceName去获得节点的所有urllist,然后通过一致性hash去获得某个具体的url

一致性hash

一致性hash解决了普通hash的问题，第一个就是如果一个节点宕机了，那么所有请求就会重新hash，然后又有可能造成某一个节点流量过大，造成雪崩效应，第二个问题就是扩容也会导致所有的请求都重新hash。

所以一致性hash就解决了这两个问题，他是通过hash环来解决的，那么hash环也有一个问题就是有可能使分布不均（数据倾斜），所以就有了虚拟节点， 每个物理节点都虚拟出多个虚拟节点，这样的话就避免了hash数据倾斜的问题

final关键字

final关键字可以用来修饰变量、方法、类。

为什么有final：因为在有些变量从设计的时候就设定了赋值之后是不允许修改的，以及为了保证线程安全。

final修饰成员变量、静态变量、局部变量

成员变量：赋值在声明的时候就赋值，或者说是构造函数中赋值，

静态变量：声明的时候赋值或者在static静态代码块赋值

局部变量：只要在使用之前赋值就可以了

final修饰方法：不允许重写重载

final修饰类：不能被继承

接口和抽象类以及区别

接口和抽象类都是为了面向对象OOP编程思想的抽象，他们都是对一个实体类的抽象

区别：

1：从设计的本身出发：接口更多的是一种行为规范，是一种功能上的共性，而抽象类更多的是一种模板式设计，是属性上的共性，就比如门有开和关两个方法，但是有的门有报警器，但有的门没有，所以这里不能直接把开和关和报警放在同一个接口或者抽象类里面，而是开和关是所有门的属性，所以需要设计一个抽象类，然后报警是个功能，所以设计为接口

2.第二就是语法和使用上的区别：接口的话因为是行为规范，所以一个类可以有多个行为，那么他就可以实现多个接口，然后抽象类因为是属性上的，所以只能单继承；在具体的语法上：接口不能实现方法，除了default方法，所以一旦实现了某个接口就需要重写所有的方法，而抽象类因为是模板式设计，所以他可以有自己的普通方法，子类可以直接调用

GC什么时候发生

GC分为minnr GC和full GC,minor GC是对新生代中的垃圾回收，当新生代中的eden区不够分配对象的时候，就会发生minor GC，minor GC采用而是复制算法，

full GC发生时期，当老年代的内存不够容不下对象的时候，就发生full GC,一般有两种情况，就是在minnor GC的时候，会首先有一个老年代担保原则，老年代剩余的可用的连续空间如果小于新生代历代minor GC升入老年代对象的总和的平均大小，就会发生一次full GC。还有就是当minnor GC之后，如果晋升的对象老年代放不下，也会full GC,

full GC可以回收方法区中的无用的常量和无用的类，无用的类的判定：没有实例对象、类加载器被回收、class对象不再任何地方被访问，以及无法通过反射访问该类，就允许被回收，但是一般方法区中的回收性价比是很低的，所以的话，可以设置虚拟机参数来确定是否回收

谈谈你对面向对象编程的理解

面向对象编程是一种将复杂问题分解为小问题，并通过封装继承多态来组织和管理代码。他将数据和行为封装为一个类，类相当于是一个模板，他描述了一组拥有共同方法的对象。

封装就是说为了隐藏内部实现的细节，提供对外访问的接口，限制外部世界直接访问，这样设计更加的安全可靠，避免外部的非法修改和破坏，同时也增加了代码的可扩展和可维护，降低了代码的耦合度，增加了代码的复用。

继承：继承就是类之间组合关系的一种，还有一种是组合，继承就是子类可以去继承父类的所有非私有的属性和方法，并且可以在自身中添加新的属性和方法。这样可以避免重复编写相同的代码，提高代码的可复用性和可维护性，并且还可以进行重写或者重载，当然父类也可以根据访问修饰符来限定子类的访问权限。组合是根据多个类组合在一起形成一个更大的类。

多态：多态就是可以根据不同的实现类，调用不用的方法

锁升级、锁消除、锁粗化

锁升级就是：锁从无锁-》偏向锁-》轻量级锁-》重量级锁的升级，为什么要这样升级呢，因为重量级锁在获取和释放的时候需要从用户态转换为内核态，效率很低，

锁消除就是:JVM在检测到某段代码不可能存在共享数据竞争的情况，那么jvm就会对锁消除，根据逃逸分析技术，如果一段代码中的对象不会逃离该线程，那么就认为是线程安全的，不用加锁

锁粗化：假设一系列的连续操作会对同一个对象反复的加锁解锁，而且在加锁解锁之间的代码能够很快的执行的话，就会把之间的代码和两段加锁的代码放在一个锁同步代码块中，避免加锁和解锁导致的性能下降，

​ stringBuffer.append 方法都需要加锁和解锁，如果JVM检测到有一连串的对同一个对象加锁和解锁的操作，就会将其合并成一次范围更大的加锁和解锁操作，即在第一次append方法时进行加锁，最后一次append方法结束后进行解锁。

session、cookie、token和jwt

因为http是无状态的协议，所以每次请求都无法知道客户端的身份，那么如果每次都登陆校验，肯定不可能，所以就有了session，第一次用用户名和密码登陆的时候，服务器端就生成一个session,存储在服务器端，然后将sessionID返回给客户端，客户端将sessionID以cookie的方式存储在客户端本地，然后以后每次访问的时候，都带着该cookie,服务器就可以快速知道身份。

这样有几个问题：（1）就是每个用户登陆之后session都存储在服务器端，登陆的用户多了就会浪费资源，（2）就是用户每次访问的服务器都需要是同一个机器，那么集群的时候就需要设置session共享。

解决办法：token,token的话就是服务器端不用存储该用户是否登陆，而是第一次登陆成功之后，就将一个token返回给客户端，然后客户端以后都带着这个token去访问服务器，然后服务器只需要校验该token是否有效。

JWT的话就是将用户的信息加密到token里面，服务器不用去查询库token是否有效，而是直接在服务器端通过密钥解析出该用户的信息，JWT由三部分组成header、payload、signature,

jwt通过非对称加密来验证了传输过程内容是否被更改。获取了header中加密算法，然后用加密算法拿着自己的私钥对payload加密，然后比对和签名是否一致，一致的话就说明没有被修改。

解决jwt过期的问题：

双token，一个accessToken一个refreshToken,首先用token 去请求资源，如果没有过期则正常进行，如果过期就带着refreshToken去获取token,

把token存放在redis中：每次访问token都去redis里面更新token过期时间

redis 大key有什么影响

大key指的是什么

  • String类型，一个String类型的数据很大，几MB
  • list、hash、set、zset中的成员个数太多或者成员的数据太大

大key产生的原因

  • 在不适用的场景使用了redis,比如使用string存放了较大的二进制数据图片文件
  • 对hash等集合没有做定时清理

如何找出大key

云redis的有DAS集成了的前三的大key,一般的话可以用redis-cli的bigkeys去获取，还有就是可以通过离线全量分析的话就是redis-rdb-tools去查看

对读

对大key进行读请求，会使redis实例的带宽被占用过大，波及其他服务

对删除

删除的话会阻塞较长时间，进而可能引发同步中断或者主从切换

对持久化

首先持久化就分为两种，aof和rdb

aof的刷盘机制有两种：同步和异步。

而AOF的刷盘时机有三种：

always：每次写了日志之后，同步将AOF日志数据写回硬盘，主线程在执行完命令之后，就调用fsync（）函数，将内核缓冲区的数据写到硬盘中，如果是大key的话，在主线程调用fsync（）的时候就会阻塞比较久的时间。

Everysec：就是每秒，主线程每次操作完命令了之后，先写到日志缓冲区中，然后每秒异步调用fsync去将缓冲区写入到硬盘中，这个的话，大key因为是异步执行所以不会影响主线程

NO：意味着不由redis控制写回硬盘的时机，而是交给操作系统

对AOF重写和RDB的影响

因为AOF重写和RDB都是通过fork一个子进程来处理任务，在创建子进程的时候，操作系统会把父进程的页表复制给子进程一份，页表记录着虚拟地址和真实物理地址的映射，所以如果是大key越来越多的话，页表的复制就会阻塞挺长时间

还有就是在父子进程修改了共享内存的数据的时候，创建完子进程后，如果子进程或者父进程修改了共享数据，就会发生写时复制，这期间会拷贝物理内存，如果内存越大，自然阻塞的时间也越长；

怎么处理

  • 对大key进行拆分
  • 定时清理
  • 即使监控内存水位

如何删除大key

选择低峰期去删除，或者分批次删除或者异步删除 unlink
